LustEngine Part 1: Project Intro

I’m writing a game engine. This isn’t usually a good idea if you want to make games, and finish them. But I’ve got a bunch of experience working on games and games middleware, and I enjoy working on that part of the stack - so this is a process I find intrinsicly valuable. If your goal is to make a game Don’t Do What Donny Don’t Does.

What is LustEngine

Lust is Love2D in Rust (filtered through my own preferences and biases). If you want Love2D in Rust, faithful to the original philosophies of Love2D, ggez is solid.

My Goals

  • Learn Rust - I’ve worked in C++ and C#, the two main game programming langauges, both professionally and at home. I enjoy them both. Rust is the new shiny, and the sales pitch is very compelling. I have no idea how well it lives up to that pitch, or how well it suits making games. I want to find out.
  • Avoid Unsafe Rust - This is a guideline I decided upon after initially diving head first in to unsafe Rust in order to help write Rust as if I were writing C. Until I’m comfortable writing Rust using the safe primitives provided by the language, I’m going to do my best to avoid unsafe code.
  • Enjoyable to Use - I’ve used a decent number of game engines and frameworks, but nothing has ever felt as good as Love2D for making small games. I want to construct APIs which are similarly quick and easy to use. This is not to say that the API will look anything like Love2D’s - it almost certainly won’t.
  • A Better Workflow - One of my biggest bugbears is awkward time consuming workflows, that distract me, that consume time, that make making games less fun. I want to make sure that the tooling and workflow makes it as quick and painless as I can make it to get my game logic where I want to be.
  • State Rewinding/Replaying - This is the main tool I want to bring to bear in creating a better workflow. So much time goes down a sink of “Get to place where unexpected behaviour happens, examine outputs, relaunch game and try a different interaction”. Maybe this is indicative of other larger workflow issues, but I really want to be able to quickly reset to the point before I experienced unexpected behaviour.
  • Determinism - Building on top of being able to rewind gamestate, I’d like to be able to reconstruct play states reliable from saved user input sessions. This would allow me to record a sample play session, make changes to the code, and quickly see if behaviour changes.
  • Target small games - I have no intentions of this engine competing with any commercial or established open source offering in terms of performance. I will happily sacrifice speed in order to facilitate more pleasant development workflows.

These Blog Posts And A Disclaimer

So I’m making some progress with this project. And I’m starting to gain insight in how and why to approach structuring code for Rust. There is a lot of trial and error, so I’m going to document my learnings here and hopefully they’ll be of use to others on similar journeys.

The massive disclaimer: I am figuring this out as I go along. I make no guarantee that I’m doing any of this the best way. But documenting my experiences may be useful.
